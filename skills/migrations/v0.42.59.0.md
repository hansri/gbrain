# v0.42.59.0 migration: multi-source ownership hardening

## Why this gate exists

The upgrade adds composite file identity `(source_id, storage_path)` and makes
page-file ownership `(source_id, source_path)` explicit. Source-qualified
object keys protect the byte plane. A legacy brain may contain two page rows
claiming one file path. GBrain must not guess which page is authoritative.

The migration therefore checks ownership before applying its schema changes.
It preserves the legacy global `storage_path` constraint during this canary so
the schema remains structurally compatible with a deliberate code rollback.
That compatibility does not authorize the previous binary to keep serving or
writing after promotion: all old services stay stopped. Removing the constraint
is deferred to a separate post-canary migration. The gate works the same way on
Postgres and PGLite and never deletes page content.

This version is a prerelease-only supervised boundary. It must not be published
as GitHub's normal `latest` release because v0.42.58-era updaters do not contain
the new local release policy. Private/fork deployments remain manual and staged.

## What the agent must do

1. Stop every GBrain service, sync/import writer, and scheduled migration. Take
   one verified recovery snapshot containing the database and the GBrain state
   roots in use (the canonical `.gbrain` directory plus any legacy
   `$GBRAIN_HOME` or `$HOME/.gbrain` copies still present).

2. Stage the new release as an immutable binary or release directory without
   starting it. The installed old binary does not contain `upgrade-preflight`,
   so the first hop into this release must use the staged **new** binary rather
   than trusting the old binary's `gbrain upgrade` wrapper. In the commands
   below, replace `/path/to/v0.42.59.0/gbrain` with that staged executable.
   Resolve and verify the exact v0.42.59.0 artifact or tag before staging it;
   do not use an unpinned package-manager `latest`, moving branch, or ambient
   `PATH` executable for any step in this cutover.

3. Run the supported preflight with the new binary:

   ```bash
   /path/to/v0.42.59.0/gbrain upgrade-preflight --json
   ```

4. If `status` is `blocked`, inspect each returned group and identify the page
   genuinely owned by that file. Keep one owner explicitly:

   ```bash
   /path/to/v0.42.59.0/gbrain upgrade-preflight repair \
     --source <source-id> \
     --path <repo-relative-path> \
     --keep <legitimate-page-slug> \
     --yes
   ```

   This clears `source_path` only on the competing rows. It preserves their
   slugs, content, versions, and audit history. If a competing row should own a
   different file, correct that page through the normal source workflow after
   the upgrade.

5. Repeat the new binary's preflight until it exits 0 with `status: ok`.

6. With services still stopped, use only the new binary to apply migrations and
   verify the result before promoting the release:

   ```bash
   /path/to/v0.42.59.0/gbrain apply-migrations --yes --non-interactive
   /path/to/v0.42.59.0/gbrain doctor
   ```

7. Confirm both ownership indexes exist. The migration orchestrator performs
   this check automatically and fails the upgrade if either index is absent.
   Only after both commands are green should the release be promoted, services
   restarted, and one `gbrain sync --all` reconciliation run with the new
   binary.

### Recovery after the handoff exists

If the binary handoff has already been published and `post-upgrade` is
incomplete, the global safety gate deliberately blocks direct mutating
`apply-migrations` and `upgrade-preflight repair` calls. Use only the bound
recovery wrappers printed by the failing command:

```bash
gbrain post-upgrade recover-migration --force-retry <exact-version>
gbrain post-upgrade repair-ownership \
  --source <source-id> --path <repo-relative-path> \
  --keep <legitimate-page-slug> --yes
```

Each wrapper acquires the canonical transition lock, rechecks the transition,
current release, and database identity, performs only the named repair, and
then resumes the normal post-upgrade gates in the same invocation. It is not a
general way to pass arbitrary migration or preflight flags.

## Rollback boundary

After the migration is fully complete and `gbrain doctor` is green, the legacy
global file conflict key leaves a short structural rollback window. It does not
permit mixed-version operation: keep every previous-binary service stopped.
Only reintroduce the previous binary during a deliberate rollback after all
writers are stopped and the database/state gates below are satisfied.

If any migration is `partial`, `wedged`, `ambiguous`, has an unresolved inflight
fence, or doctor is red, binary-only rollback is unsupported. Stop everything
and restore the matched pre-upgrade database **and** canonical/legacy GBrain
state snapshot together with the previous binary. Restoring only the executable
can make old state receipts disagree with the database.

## Do not

- Do not delete duplicate pages merely to satisfy the gate.
- Do not choose an owner by recency alone; inspect provenance and source files.
- Do not bypass the preflight with manual index creation.
- Do not substitute `bun update`, `clawhub update`, a moving Git branch, or an
  unverified download for the exact staged release.
- Do not start normal operation until the staged new binary's migration and
  doctor gates are green.
